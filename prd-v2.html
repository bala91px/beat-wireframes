Beat — Field Sales Pitch & Order App

A tablet-first tool for FMCG field reps to run a brand-controlled product pitch, capture an order with negotiated pricing, and issue a Tally-style invoice — backed by a web admin that controls the catalog, pricing and content. This v2 PRD is scoped to match the committed MVP proposal: a disciplined first release with a clear Phase-2 roadmap.

01

Product overview

The client is an FMCG manufacturer (factory in South Africa) selling an umbrella of own-manufactured brands into the Mozambique market through ~100+ field sales reps across regions, carrying ~1,000 SKUs. Beat digitises how a rep pitches products and captures orders at the counter — without changing how their team already works. The thing the client is buying is control: every rep sells the way the brand decides, and every price holds.

The core loop

Rep logs in→ Today's Beat (tasks)→ Open a catalog→ Pitch products in order→ Set qty + price→ Mandatory ad→ Save & verify→ WhatsApp invoice→ Send

It is best understood as a brand-controlled guided pitch, not a free-browse catalog. The brand dictates which products are shown, in what order, and forces ad/TVC views between them. The rep shows the tablet across the counter to the shopkeeper. Price is negotiable only down to an admin-set floor.

Scope & delivery at a glance

This section ties the PRD to the committed proposal. The MVP is deliberately disciplined; everything the client will eventually want is mapped and fenced as Phase 2, priced separately.

What the MVP is in scope

  • S1Admin console (web): products, SKU availability, branded card designer, price tiers + floors, catalog/task builder, ad gates, orders view, outlets.
  • S2Rep app (tablet PWA): login, Today's Beat, guided locked pitch, ad gates, negotiate-to-floor, order lifecycle, Tally-style invoice on WhatsApp, offline-tolerant.
  • S3UI language: English. Invoice output: Metical (MTn) + Portuguese amount-in-words + Mozambican layout (matches their Tally).
  • S4Scope: single region. One codebase (Next.js + Supabase), tablet-first Android PWA.

Fenced to Phase 2 quoted separately

  • S5Party credit ledger & outstanding (credit control), live stock / ERP sync, trade schemes (buy-X-get-Y, slabs).
  • S6Full Portuguese UI + multi-region / multi-currency structure (the invoice output is already localised in MVP; the app UI is not).
  • S7WhatsApp Business API, payment gateway / online collection, analytics, GPS / route, in-field store onboarding, legal tax-invoice compliance.

Goals & principles

Product goals

  • G1Brand controls the pitch — sequence of catalogs & products is admin-defined and enforced.
  • G2Fit the existing workflow — save, phone-check stock, in-shop approval, WhatsApp sign-off. Don't force new behaviour.
  • G3Fast order capture at the counter with negotiated, floor-protected pricing.
  • G4Everything dynamic from admin — catalog, pricing, content, ads update without a redeploy.

Build principles (for the dev)

  • P1One codebase, one API, one DB. Admin + rep app share a backend.
  • P2Data-driven, not hard-coded. Product cards, tiers, ad slots are configuration, rendered at runtime.
  • P3English UI for the MVP, but i18n-ready. Keep all UI strings externalised so a Portuguese UI can be added in Phase 2 without refactor. The invoice output is localised now (MTn + Portuguese amount-in-words).
  • P4Resilient in the field — tolerate patchy signal; never lose a drafted order.
  • P5Single-region now, region-aware in the schema. Model region/currency as data so multi-region is a Phase-2 config expansion, not a rebuild.
04

Platform & recommended stack

Rep client
Installable PWA, tablet-first (Android), responsive down to phone. No native app. Tablet is the primary screen because it is shown to the shopkeeper.
Admin client
Web dashboard (same codebase), SSR — data-heavy management UI.
Architecture
API-driven + database-backed. Admin writes config; rep PWA reads via API and caches locally. Admin changes reflect in the app in near-real-time.
Framework
Next.js + TypeScript — serves both the admin (SSR) and the installable rep PWA; shared components + API routes.
Backend / DB
Supabase — Postgres, Auth (roles/permissions), Storage (product images, videos), Row-Level Security (region/role scoping), Realtime (config push). Relational model fits regional pricing.
Invoice / PDF
Server-side PDF (React-PDF or Puppeteer). Portuguese number-to-words + weight roll-up. Localised output even though the UI is English.
WhatsApp
MVP: generate PDF → share via wa.me deep link / native share (no cost). Phase 2: WhatsApp Business Cloud API for automated sends.
Video / ads
Storage on a low-cost CDN (Supabase Storage / Cloudflare R2 / Bunny). HTML5 video with completion gating.
Hosting
Vercel (Next.js) + Supabase. Near-zero infra cost at this scale.
Why PWA, not native: the content is presentational (catalogs, branded cards, video), devices are company-controlled, and there is no app-store distribution need. A native build roughly doubles cost and maintenance for no user-facing gain here.

Rep app — feature specs

6.1 Authentication MVP

Low-friction login for field staff onboarded by admin.

  • R-A1Admin onboards a rep and issues a login code / key; rep signs in with it (no self-registration).
  • R-A2Session persists on the device; rep stays logged in across the working day.
  • R-A3Rep only sees data scoped to their assigned tasks / region.

6.2 Today's Beat (home) MVP

Landing screen = the rep's task list for the day (ref wireframe v2).

  • R-B1List of tasks assigned for today; each task maps to a catalog to present.
  • R-B2Each task shows its hero/push target where set (e.g. "push: 50 boxes").
  • R-B3Tapping a task opens the catalog and starts the guided pitch.

6.3 Guided catalog pitch MVP

A catalog is an ordered set of products (~20–25) shown one at a time. The order is brand-controlled and enforced.

  • R-C1Products are presented in the exact admin-defined order; the rep advances one product at a time.
  • R-C2A catalog may designate a hero product with a push target quantity.
  • R-C3A rep can run multiple catalogs for one shopkeeper in a single visit; all lines roll into one order.
  • R-C4Ad/TVC gates (§6.5) can sit between products and at catalog end, defined by admin.

6.4 Product pitch page MVP

Single branded product (ref wireframe v4). The visual card is composed from admin config (§7.3).

  • R-P1Shows the branded product image with overlaid logo, brand colour and price-tier badges (e.g. 1 / 100 / 250 boxes @ rates) — all rendered from data.
  • R-P2Right side: unit rate + live line amount.
  • R-P3Quantity input (boxes only). On change, amount recalculates against the applicable price tier (§9).
  • R-P4Price negotiation: rep may lower the unit price; if below the admin-set floor, the app blocks with an error and prevents adding the line.
  • R-P5Tiers & floor resolve by the shop's region (§9).
  • R-P6"Add / update line" adds the product to the running order and advances the pitch.
UOM is simplified: ordering is boxes (case-pack) + quantity only — no cartons/pallets hierarchy. The 1/100/250 figures are quantity breakpoints for pricing, not separate units. The invoice surfaces case-pack (box) qty and per-unit rate for readability. Trade schemes (buy-X-get-Y, slabs) are Phase 2 (§11).

6.5 Mandatory ad / TVC gates MVP

A control mechanism that forces the pitch sequence and guarantees ad exposure.

  • R-V1At an admin-defined gate, a video plays; the rep cannot proceed until it finishes.
  • R-V2Gate playback is non-skippable by default (admin may mark a gate skippable).
  • R-V3Only after the full duration completes does the next product / catalog unlock.
  • R-V4Gate videos are pre-cached with the beat so they play with poor signal.

6.6 Order lifecycle: save → verify → send MVP

Mirrors the client's existing manual flow — do not shortcut it.

Draft→ Saved→ Stock-checked→ Sent for approval→ Confirmed
  • R-O1Rep can save an order as a draft to review with the shop owner (and a second approver).
  • R-O2Per-line stock check — rep marks each line verified (they phone to confirm availability). Order status reflects completeness. This is a manual check; live stock is Phase 2 (§11).
  • R-O3When all lines are verified, rep generates the invoice (§6.7) and shares it on WhatsApp for approval + cost sign-off.
  • R-O4Final send / confirm records the order. Order carries status history.
  • R-O5Drafts survive connectivity loss and app restart (§10).

6.7 Invoice (Tally-style proforma) MVP

Must resemble the client's Tally "FATURA PROFORMA". Output is a localised PDF for WhatsApp — even though the app UI is English, this artifact is Mozambican.

  • R-I1Total Weight in Kgs shown at the top — sum of (product unit weight × qty) across lines.
  • R-I2Columns: SI No · Description · Quantity · Rate · per · Amount; grand total.
  • R-I3Currency Metical (MTn); amount in words in Portuguese; proforma header + authorised-signatory block.
  • R-I4Party / shop details populate from the selected outlet.
  • R-I5Generated as a downloadable/shareable PDF; not (in MVP) a legally-filed tax document — see §12.

Admin portal — feature specs

7.1 Auth, users & roles MVP

  • A-U1Admin login with role-based permissions (super admin; regional admin optional — Q6).
  • A-U2Onboard reps, issue login codes, assign region / tasks, deactivate.

7.2 Product management MVP

  • A-P1CRUD products: name, category, base image, unit weight (kg) — weight feeds the invoice.
  • A-P2Enable/disable a product per region (a SKU may sell in one region, not another).
  • A-P3SKU availability on/off (admin-controlled) — the MVP stand-in for stock visibility. A rep cannot book a SKU the admin has switched off. Live stock / ERP sync is Phase 2 (§11).
  • A-P4Bulk import of the ~1,000-SKU catalog via CSV, with a validation preview and error reporting. assumes clean CSV — Q2

7.3 Product-card designer MVP

The admin composes the branded pitch card the rep shows. Build as a data-driven template layer, not an in-browser image editor.

  • A-C1Upload product image; set brand colour and place a logo.
  • A-C2Define the price-tier badges (label + rate per tier, e.g. 1/100/250 boxes) as structured data.
  • A-C3Overlays render over the image at runtime (HTML/CSS/SVG layer) — no flattened/exported image required.
  • A-C4Live preview of the composed card before publish.

7.4 Catalog & task builder MVP

  • A-T1Create catalogs; add products; drag-reorder products within a catalog (order is enforced in the app).
  • A-T2Mark a catalog's hero product + push target quantity.
  • A-T3Assemble catalogs into tasks and assign to reps; reorder catalogs within a task.
  • A-T4Place ad gates (§7.5) between products or at catalog end.

7.5 Pricing & ad configuration MVP

  • A-R1Per product per region: define quantity price tiers and a floor price (§9).
  • A-R2Upload ad / TVC videos; set skippable flag; assign to gates.
Not in MVP: trade schemes / free-goods (buy-X-get-Y, combos), and party-level credit limits. Pricing control in MVP = tiers + floor only. Schemes and credit are Phase 2 (§11).

7.6 Orders & outlets MVP

  • A-O1View orders with status, lines, negotiated prices, weight, rep & outlet.
  • A-O2Manage the outlet / shop list reps select from (pre-registered shops). New-store onboarding = Phase 2.

09

Pricing engine — rules

  • PR1Resolve context: pricing = f(product, region). Every rate/floor lookup is region-scoped (single region for MVP, but resolved through the region key).
  • PR2Tier rate: tiers are quantity breakpoints (e.g. ≥1 → 880, ≥100 → 870, ≥250 → 860 MTn). The applicable rate is the highest breakpoint the entered quantity satisfies.
  • PR3Negotiation: rep may reduce the unit rate below the tier rate. Allowed only while rate ≥ floor_price. Below floor → hard error, line cannot be added.
  • PR4Line amount = applied_rate × qty. Line weight = unit_weight_kg × qty.
  • PR5Order totals = Σ line amounts, Σ line weights (weight drives the invoice header).
  • PR6Tiers & floor are fully admin-editable per product per region; changes reflect in the app on next sync.
  • PR7Out of scope: trade schemes / free-goods and slab-vs-whole-order discounting variants beyond simple tiers are Phase 2 (§11).
Confirm on discovery (Q3): whether tier pricing is whole-order (all units at the band rate — assumed here) or slab (each band priced separately). Affects PR2 only.
10

Non-functional requirements

Field resilience & performance

  • N1Offline-tolerant: on beat start, cache the day's catalogs, product cards, images and gate videos. Draft orders queue locally and sync when back online. depth TBD — Q1
  • N2Pitch screens must feel instant on a mid-range Android tablet; pre-load the next product.
  • N3Installable PWA (add-to-home-screen), full-screen, portrait + landscape.

Localisation, security, ops

  • N4Localisation (MVP): app UI in English; invoice output in Metical formatting + Portuguese amount-in-words. All UI strings externalised so a Portuguese UI is a Phase-2 add, not a refactor.
  • N5RBAC + region scoping enforced at the API (row-level), not just the UI.
  • N6Config changes (catalog, pricing, ads) are versioned/auditable; no redeploy needed.
  • N7Media (images/video) served from CDN; sensible upload size limits.
11

Out of scope for MVP (Phase 2+)

Named on purpose. A disciplined MVP with the full beat-system roadmap already mapped — this matches the proposal's out-of-scope band exactly, so the client sees we know what a complete system needs and chose a lean first release.

  • X1Phase 2Party credit ledger & outstanding — credit limit / balance visible at the counter before booking (the #1 margin control in FMCG trade).
  • X2Phase 2Live stock / warehouse / ERP integration — order capture is decoupled from stock in MVP (SKU on/off is the stand-in).
  • X3Phase 2Trade schemes / free-goods — buy-X-get-Y, slab discounts, combos.
  • X4Phase 2Full Portuguese UI + multi-region / multi-currency structure (invoice output is already localised in MVP).
  • X5Phase 2WhatsApp Business API automated sends (MVP uses share deep-link).
  • X6Phase 2Payment gateway / online collection — cash/credit terms handled off-app in MVP.
  • X7Phase 2Analytics — coverage %, rep productivity, top SKUs.
  • X8Phase 2GPS tracking + beat-route planning, and in-field new-store onboarding (GPS + photo + contact).
  • X9Phase 2Legal tax-invoice compliance + marketing website (budget-dependent add-ons).
12

Open questions / assumptions to confirm

Some earlier questions are now decided by the frozen proposal (localisation, region scope); the rest must be locked on the direct client call before the build starts.

  • Q1Offline depth — how patchy is field signal? Drives sync complexity (N1).
  • Q2SKU data readiness — ~1,000 SKUs as clean CSV, or scattered images/PDFs needing a data pipeline (add-on)?
  • Q3Pricing model — whole-order tier vs slab (PR2)?
  • Q4Invoice legality — is the invoice a legal Mozambican tax document, or an order slip? (Legal compliance is Phase 2 either way.)
  • Q5Data-entry model — central admin only, or regional admins editing pricing/SKUs? (Affects RBAC scope, A-U1.)
  • Q6IP ownership — bespoke (client owns) vs licensed product (we own). Kept soft in the proposal; settle in the contract before build.
Now decided (was open in v1): Localisation — English UI MVP, invoice output localised, full Portuguese UI is Phase 2. Region scope — single region for MVP, multi-region is Phase 2. Payment collection — out of app for MVP (Phase 2 gateway).
